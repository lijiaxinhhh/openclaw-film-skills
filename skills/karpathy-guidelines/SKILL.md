# Karpathy Guidelines

源自 Andrej Karpathy 对 LLM 编码陷阱的观察。每次执行任务前必须回顾这四条原则。

**权衡：** 这些原则偏向谨慎而非速度。简单任务灵活处理。

## 1. 先思考再编码

**不要假设。不要隐藏困惑。呈现权衡。**

执行前：
- 明确说出你的假设。不确定就问。
- 有多种理解时，列出来——不要默默选一种。
- 有更简单的方案就说出来。该推回就推回。
- 有不清楚的地方就停下来。说出哪里困惑。问。

## 2. 简单优先

**解决问题的最少代码。不做投机性功能。**

- 不做没要求的功能
- 不为单一用途搞抽象
- 不搞没要求的"灵活性"或"可配置性"
- 不为不可能的场景写错误处理
- 200 行能 50 行搞定就重写

自问："高级工程师会觉得这过度设计了吗？" 是就简化。

## 3. 精准修改

**只改必须改的。只清理自己制造的垃圾。**

编辑现有内容时：
- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你会做得不同
- 发现不相关的死代码，提出来——不要删除

你的修改产生孤儿代码时：
- 删除你的修改导致未使用的导入/变量/函数
- 不要删除之前就存在的死代码，除非被要求

检验：每一行改动都能直接追溯到用户的要求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

把任务转化为可验证的目标：
- "加验证" → "为无效输入写测试，然后让它通过"
- "修 bug" → "写一个复现测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务，列出简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强成功标准让 LLM 能独立循环。弱标准（"搞定它"）需要不断澄清。

## 反模式总结

| 原则 | 反模式 | 修正 |
|------|--------|------|
| 先思考 | 默默假设文件格式、字段、范围 | 明确列出假设，询问确认 |
| 简单优先 | 单一折扣计算用策略模式 | 一个函数直到真正需要复杂度 |
| 精准修改 | 修 bug 时顺手改引号风格、加类型注解 | 只改修复问题的行 |
| 目标驱动 | "我来审查改进代码" | "写测试复现 bug → 让它过 → 验证无回归" |

## 核心洞察

过度设计的代码不是明显错误——它们遵循设计模式和最佳实践。问题是**时机**：在不需要时增加复杂度，会让代码更难理解、更容易出bug、更难测试。

好的代码是简单地解决今天问题的代码，而不是过早解决明天问题的代码。
